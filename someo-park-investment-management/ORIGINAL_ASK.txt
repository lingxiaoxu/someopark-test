1. 设计目标

在 Someo Park 官网增加一个“小窗式接入向导”，允许用户将自己的 OpenClaw 接入到 Someo Park 平台，并调用 Someo Park 已有量化策略能力，包括：
	•	策略回测
	•	Walk-forward 验证
	•	每日信号生成
	•	持仓监控
	•	生产交易前的执行建议
	•	后续扩展到 paper trading / live trading

这个设计必须以现有代码库为基础，而不是重新发明一套策略系统。当前仓库已经具备双策略框架、参数集、配对筛选、每日信号和 walk-forward 体系：README 写明项目为“双策略配对交易回测框架”，包含 MRPT 与 MTFS 两个并列策略，并共享统一基础设施 。README 和 RUNBOOK 还明确列出了 DailySignal.py、SelectPairs.py、MRPTWalkForward.py、MTFSWalkForward.py、PortfolioMRPTStrategyRuns.py、PortfolioMTFSStrategyRuns.py 等核心脚本   。

2. 当前代码库能力抽象

根据仓库文档，现有系统不是单纯的研究代码，而是已经接近一个可运行的策略引擎，至少包含以下六层能力：

2.1 数据与环境层

系统依赖 .env 中的 POLYGON_API_KEY、FRED_API_KEY、MONGO_URI、MONGO_VEC_URI，并统一要求在 someopark_run conda 环境下执行，说明当前引擎已经依赖外部市场数据、宏观数据和数据库资源   。

2.2 策略层

仓库当前有两条主策略线：
	•	MRPT：均值回归配对交易，基于 Kalman Filter 对冲比率与动态 z-score 逻辑
	•	MTFS：动量趋势跟随配对交易，基于多窗口动量评分、SMA 趋势确认和波动率调仓

2.3 研究与优化层

现有仓库已经支持：
	•	grid search
	•	DSR 选参
	•	Step1 → Step2 → Step3 三步法
	•	walk-forward 多窗口 OOS 验证
	•	IS/OOS 报告生成

这些流程在 README 与 RUNBOOK 中都有完整运行说明   。

2.4 日常运行层

README 明确写到 DailySignal.py 用于每个交易日收盘后运行，输出 OPEN / CLOSE / HOLD / FLAT 信号，并生成报告、更新 inventory、执行 regime 检测和 position monitor 。RUNBOOK 还给出了 --strategy both、--dry-run、--skip-regime、--date 等生产级命令入口，说明日常运行层已经基本成型  。

2.5 持仓状态层

仓库当前已经有：
	•	inventory_mrpt.json
	•	inventory_mtfs.json
	•	inventory_history/

并记录开仓时间、价格、param_set、hedge_ratio、open_signal、wf_source 等字段，用于 Position Monitor 精确复现仓位状态   。

2.6 风控与环境自适应层

README 写明 DailySignal.py 内含 Regime 检测，综合波动率、信用利差、利率、AI 动量等指标给出 0–100 分，从而动态调整 MRPT 与 MTFS 的资金权重；同时 Position Monitor 会以开仓时参数持续监控止损与平仓条件 。

因此，SomeoClaw 的设计原则不是“把 OpenClaw 变成策略引擎”，而是：

OpenClaw 负责接入、触发、编排和对话交互；Someo Park 现有代码库继续负责核心量化逻辑。

3. SomeoClaw 的定位

SomeoClaw 是 Someo Park 网站上的一个接入层。它的功能不是重写策略，而是把用户侧的 OpenClaw 变成一个可以安全调用 Someo Park 策略服务的远程 agent。

3.1 用户视角

用户完成接入后，可以在 Someo Park 网站或其 OpenClaw 中发起以下动作：
	•	对某个策略做回测
	•	对一组参数做 walk-forward
	•	查询今日信号
	•	查询持仓监控结果
	•	生成当日执行建议
	•	后续扩展到 paper/live trading

3.2 平台视角

Someo Park 对外暴露的是 skill 接口，不是仓库内部脚本本身。也就是说：
	•	PortfolioMRPTStrategyRuns.py 不直接暴露给用户
	•	DailySignal.py 不直接暴露为 shell 命令
	•	用户只看到 skill，例如：
	•	backtest_mrpt
	•	backtest_mtfs
	•	run_daily_signals
	•	monitor_positions
	•	run_walk_forward
	•	generate_execution_plan

10. 运行层设计

10.1 执行方式

服务端执行层不应让前端直接 shell 调度，而应通过一个内部 Job Runner 实现。

每个 skill 请求进入后：
	1.	生成任务记录
	2.	写入参数
	3.	调用内部 runner
	4.	runner 以受控方式运行仓库脚本
	5.	收集输出文件
	6.	返回结构化结果

10.2 Job Runner 封装原则

因为 RUNBOOK 已明确要求所有命令都必须：
	•	在指定目录下
	•	加载 .env
	•	用 conda run -n someopark_run --no-capture-output python ...

所以内部 runner 必须统一封装这个调用规范，不能让每个 skill 自己拼命令  。

12. 风控设计

12.1 技术风控
	•	每个 agent 独立 token
	•	skill 白名单
	•	任务级 timeout
	•	并发限制
	•	审计日志

12.2 策略风控

SomeoClaw 不改动现有策略逻辑，而是在其上方加治理层：
	•	不允许未授权用户运行 live trading
	•	默认只开 dry_run 或 paper trading
	•	对高风险任务增加审批开关
	•	每个用户可设每日资本上限
	•	每个用户可限制资产范围

12.3 生产交易风控

MVP 阶段建议只做到：
	•	生成交易建议
	•	不自动下真实单

等后续再增加 broker adapter。

15. 关键结论

根据你仓库当前结构，SomeoClaw 最合理的实现方式不是“把 OpenClaw 变成策略计算器”，而是：

把 OpenClaw 变成 Someo Park 的远程 agent 接入层；把你现有 Python 双策略系统封装成可控的策略 skill 服务。

这样做有三个直接好处：

第一，你现有仓库的研究与生产逻辑可以继续复用，因为 README 和 RUNBOOK 已经表明它具备较成熟的回测、walk-forward、每日信号和 inventory 体系   。
第二，核心策略 IP 仍留在 Someo Park 服务器，而不是暴露在用户侧 OpenClaw。
第三，未来从 backtest 扩展到 paper / live trading 时，系统边界会更清晰。

# 1. 设计原则（Design Principles）

1. **AI-first interaction**

   * Chat 是 primary interface
   * 所有功能通过 conversation 驱动

2. **Low visual noise**

   * 无强边框
   * 依赖层级与背景区分

3. **State visibility**

   * 用户必须看到 agent 在“工作”

4. **Data-native UI**

   * JSON / 表格 / 图表是一级组件，不是附属
